# Research: Landing page para app de gestión de calendario con menú de navegación

**Date**: 2026-03-12
**Git Commit**: e06e4133ee788b58f2f70232cebf89758206209a
**Branch**: main

## Research Question
Quiero sustituir el landing page que traemos por una landing page donde hablemos de este proyecto que sea sobre una página para gestionar tu calendario. Quiero que tenga un menú superior o de hamburguesa donde podamos luego meter opciones. Actualmente solo pondremos una de login (esta función se evaluará más adelante).

## Summary

La landing page actual (`app/page.tsx`) es el scaffold por defecto de `create-next-app`: muestra logos de Next.js y Vercel con links a documentación y un botón de deploy — sin relación con el producto. El layout global (`app/layout.tsx`) usa el título y descripción por defecto ("Create Next App"). No existe ningún componente de navegación, header o menú en el proyecto; el directorio `src/components/` no existe todavía. Tailwind v4 (sin librerías de componentes adicionales) es el único sistema de estilos disponible, lo que significa que el menú de hamburguesa y la navbar deberán construirse con HTML/CSS vanilla + Tailwind, requiriendo `"use client"` para el toggle del menú móvil.

- Importa `next/image` para logos de Next.js (`/next.svg`) y Vercel (`/vercel.svg`) desde `public/`

- No exporta objeto `metadata` — hereda el título por defecto del layout
- Este archivo **será reemplazado completamente**

### Estado actual de `app/layout.tsx`

### Requisitos técnicos para el menú

El menú de hamburguesa requiere estado interactivo (open/closed), lo que implica:
- Un componente con `"use client"` para manejar el toggle con `useState`
- El navbar puede vivir en `src/components/Navbar.tsx` (siguiendo la estructura del CLAUDE.md: `src/components/` como ubicación para componentes)
- Puede ser incluido directamente en `app/layout.tsx` como elemento global sobre `{children}`, o incluido en `app/page.tsx` como parte de la landing

## Key Architectural Decisions Found

- **App Router de Next.js 15**: los layouts son Server Components por defecto; para interactividad (toggle hamburguesa) se necesita `"use client"` en el componente específico
- **Tailwind v4**: usa `@import "tailwindcss"` en lugar de directivas separadas; los tokens custom se definen con `@theme inline`
- **Sin componentes existentes**: no hay patrones previos de componentes en `src/components/` que seguir — será el primer componente del proyecto
- **Fuentes**: Geist Sans ya está configurada globalmente via `--font-geist-sans` → `font-sans` en Tailwind
